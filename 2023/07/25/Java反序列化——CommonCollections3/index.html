Java反序列化——CommonCollections3

Created2023-07-25|Updated2024-03-27|Java
|Word count:2.3k|Reading time:11min|Post View:

前言

分析完CC1和CC6，继续来看CC3的链子，相比较与前两者，CC3改变了最后的执行类，由Runtime命令执行改变为动态类加载，从而能执行任意代码，下面就是对其具体分析。

动态类加载

ClassLoader 是什么呢？它就是一个“加载器”，告诉Java虚拟机如何加载这个类。Java默认的 ClassLoader 就是根据类名来加载类，这个类名是类完整路径，如 java.lang.Runtime 。

ClassLoader类加载过程

ClassLoader继承关系：
ClassLoader => SecureClassLoader => URLClassLoader => AppClassLoader

ClassLoader类加载调用过程：
loadClass => findClass => defineClass(从字节码加载类)

URLClassLoader.loadClass()任意类加载

URLClassLoader可以通过多种协议（file/http/jar）远程或本地实现任意类加载。

ClassLoader.defineClass()字节码加载任意类

ClassLoader.defineClass()可以通过加载字节码实现任意类加载，但defineClass是protected的，只能通过反射调用。

与URLClassLoader相比，defineClass是protected的方法，如果要放到反序列化场景下，是不能直接反射调用触发的，所以我们还需要寻找哪些public的方法调用了defineClass，来构造攻击链。

Unsafe.defineClass()字节码加载任意类

Unsafe是一个比较底层的类，其defineClass是public的，但是该类却不能直接实例化（安全检查），所以仍需要反射来实例化。但在Spring里类可以直接实例化。

CC3链具体分析

利用 TemplatesImpl 加载字节码

在实际场景中，因为defineClass方法作用域是不开放的，所以攻击者很少能直接利用到它。我们需要找到一个重写了defineClass方法并且能够被类外部调用，这就是 TemplatesImpl 。

com.sun.org.apache.xalan.internal.xsltc.trax.TemplatesImpl 这个类中定义了一个内部类 TransletClassLoader ：

这个类里重写了 defineClass 方法，并且这里没有显式地声明其定义域。Java中默认情况下，如果一个方法没有显式声明作用域，其作用域为default。所以也就是说这里的 defineClass 由其父类的 protected类型变成了一个default类型的方法，可以被类外部调用。

注意，这里有三个地方调用了 TemplatesImpl#defineTransletClasses()，选择 TemplatesImpl#getTransletInstance() 是因为我们需要类对象进行实例化。

这与 defineClass 的特性有关，在 defineClass 被调用的时候，类对象是不会被初始化的，只有这个对象显式地调用其构造函数，初始化代码才能被执行。而且，即使我们将初始化代码放在类的static块中，在 defineClass 时也无法被直接调用到。

TemplatesImpl#getTransletInstance() 中，在调用完 TemplatesImpl#defineTransletClasses() 后又调用了newInstance()构造函数，完成了初始化，触发了代码执行。
